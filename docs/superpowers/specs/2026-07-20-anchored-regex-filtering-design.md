# 锚点门控正则过滤设计

## 目标

在不改变既有 `keywords.txt` 存储格式、管理接口或字面量匹配性能的前提下，增加用于识别结构化敏感信息的正则规则能力。规则必须先命中指定的精确字符串锚点，才会进入正则匹配阶段。

请求体最大为 1 MiB。设计应避免逐条 PCRE 扫描请求体所造成的 `规则数 x 请求体大小` 成本。

## 非目标

- 不把正则语法写入 `keywords.txt`。
- 不改变现有 `/keywords` API、关键词 CLI 子命令或关键词文件的一行一个字面量语义。
- 不支持任意 PCRE 特性。回溯、反向引用和递归等无法给出稳定资源边界的特性不纳入规则语言。

## 配置

### 既有关键词

`/etc/openresty/keywords.txt` 保持不变。每个非空行都是一个字面量关键词，并继续由 Aho-Corasick 自动机匹配。

### 正则规则

新增可选文件 `/etc/openresty/regex_rules.jsonl`，每行一个 JSON 对象：

```json
{"id":"contract-total-price","anchor":"服务总价","expression":"{{anchor}}\\s*[（(]?\\s*含税\\s*[)）]?\\s*[:：]\\s*[￥¥]\\s*\\d{1,3}(?:,\\d{3})*(?:\\.\\d{1,2})?","enabled":true}
```

字段约束：

- `id`：唯一、稳定的 ASCII 标识符，用于审计日志、管理 API 和命中记录。
- `anchor`：必填的精确字符串，至少 6 字节。应选择低频且与目标信息稳定共现的文本。
- `expression`：必填的受限正则表达式，必须至少包含一次 `{{anchor}}` 占位符。加载时将其替换为经过正则转义的 `anchor`。
- `enabled`：可选，默认为 `true`。

锚点不是仅供优化的提示。规则语义规定目标文本必须同时满足该锚点和完整表达式，因此不会因“请求的两个无关位置分别出现锚点与表达式”而误命中。

`regex_rules.jsonl` 不存在时表示未启用正则规则，现有部署继续正常运行。

## 运行时架构

### 加载和重载

每个 OpenResty worker 维护只读的策略快照：

1. 读取 `keywords.txt`，按现有方式构建字面量匹配器。
2. 读取并校验 `regex_rules.jsonl`。
3. 收集所有已启用规则的 `anchor`，构建锚点匹配器。
4. 用受限多规则正则引擎编译所有 `expression`，得到一个 worker 本地的正则数据库。
5. 所有构建成功后，原子替换旧快照。

后续实现应使用支持多规则单次扫描且具有稳定资源界限的引擎，例如 Hyperscan。不能以循环调用 `ngx.re.find` 的方式实现规则集。

关键词和正则规则维持独立的源文件与管理接口。内存快照可以在一次 Aho-Corasick 扫描中同时识别字面量关键词和锚点，但这不改变两类规则的存储语义。

管理端新增独立的 `/regex-rules` API 和对应 CLI 子命令；`/keywords` 保持原样。对正则规则的新增或修改先在内存中校验和编译，失败则拒绝写入或回滚文件，继续使用最后一个有效快照。

### 请求路径

```text
请求体
  -> Aho-Corasick 扫描字面量关键词和正则锚点
  -> 命中字面量关键词：立即拦截
  -> 未命中任何锚点：放行，不执行正则引擎
  -> 命中至少一个锚点：多规则正则数据库扫描一次
  -> 命中正则规则：拦截；否则放行
```

为达到上述行为，现有 `ahocorasick.match` 的“返回第一个偏移量”接口需要扩展为一次扫描同时报告：

- 首个命中的字面量关键词（用于立即拦截）。
- 是否至少命中过一个正则锚点。

该接口不需要为每个命中分配 Lua 表或返回全部命中结果。它只维护两个结果状态，因此即使请求中重复出现锚点，空间开销也保持常数级。

## 性能和资源边界

- 未配置正则规则时，请求路径与当前实现相同。
- 已配置正则规则但请求不含锚点时，只进行一次 Aho-Corasick 扫描。
- 命中锚点时，对不超过 1 MiB 的原始请求体额外进行一次多规则正则扫描；不复制请求体，也不逐条扫描。
- 正则规则数量、单条表达式长度、总表达式长度和编译后数据库字节数均设置配置上限。超过上限的更新被拒绝。
- 每个 worker 持有一份正则数据库，因此继续推荐 `WORKER_PROCESSES=1`，并在健康元数据中暴露规则数量和数据库大小。

即使管理员选择了频繁出现的锚点，最坏路径也限定为一次 AC 扫描加一次多规则正则扫描，而不是按规则数量重复扫描 1 MiB 请求体。最小锚点长度仍有助于让常规请求跳过第二阶段。

## 安全和失败行为

- 表达式使用引擎支持的线性时间受限语法；不支持的语法在加载阶段报错。
- 初始加载时，如果已配置的正则规则无法读取、解析或编译，策略处于失败状态并沿用现有 fail-closed 行为。
- 运行中的规则更新失败时保留最后一个有效快照，管理调用返回错误；不得让部分规则生效。
- 正则扫描引擎返回资源或运行时错误时，拦截请求并记录策略错误，避免在检测不确定时放行。
- 日志和对外响应记录命中的 `id`，不记录完整表达式或请求体内容。

## 可观测性

`/health` 与新增的规则管理接口增加独立元数据：

- `regex_rules_status`
- `regex_rules_loaded`
- `regex_rules_version`
- `regex_rules_last_loaded_at`
- `regex_rules_load_error`
- `regex_database_bytes`

原有关键词元数据字段保持不变。命中日志应区分 `literal` 与 `anchored_regex`，并只记录关键词文本或规则 `id`。

## 测试

1. 既有 `keywords.txt` 和 `/keywords` 行为不变，字面量命中仍立即拦截。
2. 正则规则文件不存在时，网关正常启动且不进入正则阶段。
3. 锚点缺失时，1 MiB 请求不调用正则扫描器。
4. 锚点存在且表达式匹配时，返回拦截响应并记录规则 `id`。
5. 锚点存在但表达式不匹配时，请求放行。
6. 多条规则同时存在时，正则引擎只接收一次请求体扫描。
7. 非法 JSON、重复 `id`、短锚点、缺少 `{{anchor}}`、不支持的表达式和超出资源上限的更新均被拒绝，旧快照仍可用。
8. 启动时有效规则加载失败、运行时正则引擎错误均按 fail-closed 拦截。
9. 基准测试覆盖 1 MiB 的无锚点、仅锚点、命中和不命中请求，验证扫描次数与延迟上界不随规则数量线性增长。
